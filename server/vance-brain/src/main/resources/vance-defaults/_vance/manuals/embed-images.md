# Embedding вҖ” Images

How to show a picture in the chat. There are three sources, all
delivered through Markdown вҖ” no special protocol, no tool gymnastics.

## When to use this

User asks to *see* a photo, screenshot, diagram or any visual.
Examples: "zeig mir Bilder von Lissabon", "show me the logo",
"what does X look like", or you just ran `web_search` and the
results contain image URLs.

## Three ways to embed an image

### 1. External URL вҖ” most common

Standard Markdown image syntax. The Web-UI's Markdown renderer
permits `<img>` tags from `https://` (DOMPurify default profile),
so this just works:

```
![Alfama district, Lisbon](https://example.com/alfama.jpg)
```

How to find the URL вҖ” pick the right tool:

- **`image_search`** (preferred for "show me pictures of X"
  requests). Each returned `imageUrl` is pre-validated against
  the source host, so dead links never reach the user. The result
  also carries `thumbnailUrl`, `source`, `sourceLink`. Drop
  `imageUrl` straight into the Markdown above.
- `web_search` returns page hits, not image URLs. If you extract
  an image URL out of a snippet manually, it has not been
  validated вҖ” `image_search` is the safer path.
- Public references the user names directly (a Wikipedia article,
  a stock-photo URL they pasted) вҖ” use as-is.

Rules:

- Use `https://` (not `http://`). The sanitiser allows both, but
  HTTPS is the only one worth typing.
- Put a meaningful alt text in the brackets вҖ” it doubles as the
  caption in voice/TTS clients and as accessibility text.
- One image per line. Do not glue them into a single paragraph.
- Don't fabricate URLs. If you don't have a real URL from
  `image_search` or the user, don't invent one вҖ” search first or
  ask the user.

### 2. Project Document вҖ” when the image lives in this workspace

For an image already stored as a Document in the project (uploaded
by the user, generated by `image_generate`, saved earlier), use
the `document_link` tool вҖ” never hand-construct the URI:

```
[document_link(path="diagrams/architecture.png")] вҶ’
  { markdownLink: "![Architecture](vance:/diagrams/architecture.png?kind=image)" }
```

The tool returns the ready-to-paste `markdownLink` string with the
correct `vance:` URI and `?kind=image` hint. Insert it verbatim.

Tools that **already return** a `markdownLink` field in their
response (use it directly, no extra `document_link` call):

- `image_generate` вҖ” returns `{ path, markdownLink, вҖҰ }`
- `doc_create(kind="image", вҖҰ)` вҖ” same shape
- `document_create`, `document_save` вҖ” same shape

### 3. SVG inline вҖ” diagrams you draft

For small SVG you draft directly (icons, simple diagrams), inline
fence works:

```
\`\`\`svg
<svg xmlns="http://www.w3.org/2000/svg" width="100" height="100">
  <circle cx="50" cy="50" r="40" fill="steelblue"/>
</svg>
\`\`\`
```

For anything bigger or that the user wants to keep around: save as
Document first (`doc_create(kind="image", вҖҰ)`), then embed
the returned link.

## What not to do

- **Never** say "I cannot display images" without checking this
  manual. The UI renders them. The bug is the refusal, not the
  capability.
- Never hand-construct `vance:` URIs. The tool layer owns that
  format вҖ” your guess will rot the first time the schema evolves.
- Never paste raw `data:image/...;base64,...` URLs into chat. They
  are huge and the renderer treats them as binary blobs; save as
  Document instead.
- Do not wrap the image link in an `arthur_action` payload as raw
  text. Emit it through the action's `message` field like any
  other reply content.
