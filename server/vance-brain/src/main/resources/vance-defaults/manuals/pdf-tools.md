# Tool family — `pdf_read`, `pdf_metadata`

Two read-side tools for PDF documents that are already stored in
the project (uploaded by the user, generated by another tool, …).
Both use PDFBox under the hood, no external binary, no OCR.

## When to use this

- "Fasse mir das PDF zusammen" → `pdf_read` then summarise.
- "Was steht auf Seite 3?" → `pdf_read` with `pages="3"`.
- "Wer hat das PDF erstellt, wann?" → `pdf_metadata`.
- "Wie viele Seiten hat das?" → `pdf_metadata` (cheap probe before
  a full read).
- "Extract action items from this 100-page report" → `pdf_metadata`
  first to see the size, then `pdf_read` with a range or in chunks.

Counterparts in the manual library:

- `doc_read` — for plain-text documents (markdown, json, yaml, …).
  PDF-mime documents are **rejected** by `doc_read`; use `pdf_read`.
- `pdf_search` — Google search for PDFs *on the web*, not in the
  project. Different tool, different scope.

## pdf_read

### Parameters

| Param | Type | Required | Notes |
|---|---|---|---|
| `path` | string | one of | Document path inside the project |
| `id` | string | one of | Mongo id of the document |
| `projectId` | string | no | Project name; defaults to active project |
| `pages` | string | no | 1-based page range. Examples: `"3"`, `"1-5"`, `"7-"`, `"-4"`. Default: whole PDF. End beyond doc length is silently clamped |

### Returns

```
{
  id:             "...",
  path:           "papers/smith-2024.pdf",
  title:          "Computational Approaches to ...",   // when present
  pageCount:      24,
  requestedPages: "all" | "1" | "2-5",
  contentLength:  18342,                                // full chars before truncation
  truncated:      false,
  content:        "<extracted text, newline-preserved>"
}
```

Content is truncated past 50 000 characters; `contentLength`
reports full length.

### Examples

**Read whole PDF**:
```
pdf_read(path="papers/smith-2024.pdf")
```

**Read just chapter 3 (assumes you know which pages)**:
```
pdf_read(path="papers/smith-2024.pdf", pages="14-22")
```

**Read first 5 pages — typical "what's this about" probe**:
```
pdf_read(path="papers/smith-2024.pdf", pages="1-5")
```

**Read from page 10 to the end**:
```
pdf_read(path="papers/smith-2024.pdf", pages="10-")
```

## pdf_metadata

### Parameters

Same identifier params as `pdf_read` (`path` or `id`, optional
`projectId`). No `pages` option — metadata is one shot.

### Returns

```
{
  id:               "...",
  path:             "papers/smith-2024.pdf",
  sizeBytes:        2_456_789,
  pageCount:        24,
  title:            "Computational Approaches ...",   // optional
  author:           "Jane Smith",                      // optional
  subject:          "...",
  keywords:         "...",
  creator:          "Word",                            // app that authored it
  producer:         "macOS PDF Engine",                // app that wrote bytes
  creationDate:     "2024-09-12T14:32:00Z",
  modificationDate: "2024-09-13T08:11:00Z"
}
```

Only fields actually present in the PDF are returned (PDF spec
makes all of them optional). Missing fields are **omitted**, not
nulled — keeps the response tight.

### Example

```
pdf_metadata(path="papers/smith-2024.pdf")
```

Use this **before** `pdf_read` when you don't know how big the PDF
is. A 200-page paper read whole would burn a lot of tokens; the
metadata + a smart `pages` choice keeps cost in check.

## Anti-patterns

- **Don't use `doc_read` for PDFs.** It treats the bytes as UTF-8
  and you'll get garbage. `pdf_read` exists for this reason and
  rejects non-PDF mimes explicitly.
- **Don't read a 200-page PDF whole if you only need page 5.**
  Use `pdf_metadata` first to see the page count, then `pdf_read`
  with a `pages` range.
- **Don't expect OCR.** Scanned PDFs without an embedded text
  layer return empty content. A scanned-pdf detector + OCR tool
  is on the roadmap; until then, ask the user for a text-bearing
  PDF.
- **Don't expect images, figures, formulas to be preserved.**
  PDFBox extracts the text layer only — images and inline math
  become whitespace. For figure-heavy papers, the user should
  paste relevant pages as screenshots into the chat (the vision-
  capable LLM provider will handle them) and use `pdf_read` only
  for prose context.
- **Don't fabricate citations from metadata.** A PDF's `author`
  field is what the producing tool wrote there — sometimes correct,
  often "Microsoft Word User" or empty. For real citations read
  the first page or the references section explicitly.

## Failure modes

| Symptom | Likely cause | Recovery |
|---|---|---|
| "Document '...' is not a PDF" | Wrong mime type (markdown, plain text, …) | Switch to `doc_read` |
| "Invalid pages range '...'" | Malformed `pages` parameter | Use the documented shapes (`"3"`, `"1-5"`, `"7-"`, `"-4"`) |
| "PDF text extraction failed: ..." | Corrupted PDF, encrypted PDF without password | Tell the user honestly; possibly request a different file |
| Empty `content` despite OK exit | Scanned PDF, image-only layout | OCR isn't supported yet — ask user for text-bearing PDF |

## Not in this iteration

- **OCR** for scanned PDFs / image-only PDFs (Tesseract via Tess4J
  is the future path — separate `ocr_image` and OCR-fallback in
  `pdf_read`).
- **Table extraction** with structure preservation (Tabula-Java
  could land here as `pdf_extract_tables` → records-Document).
- **PDF manipulation** (merge, split, watermark, encrypt) —
  PDFBox can do all of it, separate tool family.
- **PDF generation** from Markdown — that's the Output spur,
  tracked in `planning/web-office-suite.md`.
